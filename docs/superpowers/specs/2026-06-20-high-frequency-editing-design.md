# 高频编辑体验设计

## 目标

让浮生录在高频编辑时更像一个可靠的本地创作工具。用户在整理人物、事件、关系、资料、分析笔记和图谱位置时，应能快速撤销、重做、确认保存状态、搜索当前项目，并用不同图谱布局视角理解资料。

一期目标不是重写编辑系统，而是在现有项目数据流上补齐最常用的编辑保障和导航能力。

## 背景

当前应用已经有 React、TypeScript、Vite、Zustand、React Router、React Flow、Express 本地 API 和桌面端发布流程。项目数据修改主要集中在 Zustand store 的 `commitProject(projectId, updater, options)` 路径，图谱位置、实体、事件、关系、资料库、分析笔记等编辑也通过这一层保存。

项目布局中已经有顶部搜索输入的视觉位置，但没有实际搜索能力。图谱工作台已经有 React Flow 画布、自动布局、节点位置保存、筛选、详情面板和布局保存状态的局部反馈，但这些能力还没有形成全局的高频编辑体验。

## 推荐方案

采用项目级历史层。

在 `commitProject` 外围记录每次项目编辑的前后项目快照，按项目维护撤销栈和重做栈。撤销时恢复到编辑前项目，重做时恢复到编辑后项目，并继续走现有保存同步链路。

这个方案保留现有 action 结构，不把每个编辑操作重写成命令模式。它能覆盖主要项目编辑操作，风险比命令模式重构低，也比只做图谱局部撤销更一致。

## 范围

一期覆盖：

- 撤销和重做主要项目编辑操作。
- 保存状态的项目级反馈。
- 当前项目内全局搜索。
- 图谱布局视图切换和应用。

不覆盖：

- 跨项目搜索。
- 重启后保留撤销历史。
- 多用户协作冲突解决。
- 云同步。
- 复杂物理仿真布局。
- 全量命令模式重构。

## 撤销和重做

### 用户体验

顶部项目栏提供撤销和重做图标按钮，禁用态清楚可见。快捷键支持：

- `Ctrl+Z` 撤销。
- `Ctrl+Y` 重做。
- `Ctrl+Shift+Z` 重做。

撤销和重做按当前项目隔离。切换项目后，只操作当前项目的历史栈。每个项目默认保留最近 50 条历史记录。

### 历史记录规则

所有改变项目内容的 action 默认进入历史栈，包括：

- 新增、更新、删除实体。
- 新增、更新、删除事件。
- 新增、更新、删除实体关系。
- 新增、更新、删除事件链接。
- 更新图谱节点位置。
- 批量应用图谱布局。
- 新增、更新、删除资料库条目。
- 更新分析笔记。
- 批量导入、恢复示例、清空项目数据。

以下变化不进入历史栈：

- 加载项目。
- 服务端同步回写。
- 主题切换。
- 侧边栏折叠和宽度调整。
- 后端状态轮询。
- 撤销和重做自身的恢复写入。

撤销后如果发生新的项目编辑，清空该项目的 redo 栈。

### 合并策略

连续高频操作需要合并，避免一次拖拽产生大量历史记录。

一期合并以下操作：

- 同一节点在短时间内连续拖拽，合并为一条“移动节点”历史。
- 同一次批量布局的多个节点位置更新，合并为一条“应用布局”历史。
- 同一文本字段的连续编辑如果通过明确保存按钮提交，记录为一次编辑；如果当前表单是逐字保存，则使用字段级 `mergeKey` 和 1200 毫秒合并窗口。

历史记录包含标签，用于后续显示或调试，例如“更新人物”“删除事件”“移动节点”“应用布局”。

## 保存状态

### 用户体验

顶部状态区显示当前项目保存状态：

- 保存中。
- 已保存。
- 离线保存。
- 保存失败。

保存失败时展示简短错误提示，并保留重试入口。离线时继续允许本地编辑，状态显示为“离线保存”，不阻断用户继续工作。

### 状态模型

按项目维护保存状态：

- `idle`: 尚无需要展示的状态。
- `saving`: 本地项目已变更，正在同步到后端。
- `saved`: 最近一次同步成功。
- `offline`: 后端不可用，本地已保留变更。
- `error`: 最近一次同步失败。

状态还应包含：

- `lastSavedAt`。
- `errorMessage`。

普通编辑、撤销、重做、应用布局都触发同一套保存状态变化。撤销和重做不会绕过保存链路。

## 全局搜索

### 用户体验

顶部搜索入口升级为当前项目搜索。用户可以点击输入框，也可以用 `Ctrl+K` 打开命令面板式搜索弹层。

搜索结果按类型分组：

- 实体。
- 事件。
- 关系。
- 资料。
- 笔记。

每条结果显示名称、类型、简短上下文。点击结果后跳转到对应页面；如果结果能定位到图谱节点，则进入对应图谱并聚焦节点。

### 搜索规则

搜索只查当前项目。关键词为空时不展示项目结果，只显示简短提示，不展示错误状态。

匹配字段包括：

- 实体名称、别名、类型、标签、描述、阵营、地点。
- 事件标题、描述、时间、地点、标签、参与实体。
- 关系类型、描述、起止时间、两端实体名称。
- 资料标题、作者、来源、摘要、标签、笔记。
- 分析笔记标题和正文。

每类结果最多展示 8 条。命中关键词需要高亮，大小写不敏感。中文文本按子串匹配。

## 图谱布局视图

### 用户体验

图谱页在现有工具条中新增“布局视图”入口，不新增独立页面。

一期提供四种视图：

- 自由布局：保留用户手动摆放。
- 人物关系：突出人物、组织、地点和实体关系。
- 时间事件：按时间顺序组织事件，关联人物作为辅助节点。
- 证据资料：突出资料、引用和分析线索。

切换视图先更新预览状态，不直接覆盖用户手动位置。用户点击“应用布局”后，新的节点位置写入项目，并生成一条历史记录。一次撤销应能恢复应用布局前的位置。

### 布局规则

一期采用可预测的规则布局，不引入复杂力导向仿真。

人物关系视图优先按实体类型、阵营或地点分组。时间事件视图优先按年份或顺序排列事件。证据资料视图优先让资料和被引用对象形成清晰层次。

布局输出应稳定：同一份数据在同一视图下重复计算，应得到一致或接近一致的位置，避免画布无意义跳动。

## 架构

### Store 历史层

在 Zustand store 中新增项目历史状态：

- `undoStacksByProjectId`。
- `redoStacksByProjectId`。
- `saveStatusByProjectId`。

`commitProject` 新增历史选项：

- `historyLabel`。
- `skipHistory`。
- `mergeKey`。
- `mergeWindowMs`。

普通 action 不需要大规模重写，只需要在重要编辑入口传入更准确的 `historyLabel` 和必要的 `mergeKey`。

撤销和重做通过新 action 暴露：

- `undoProject(projectId)`。
- `redoProject(projectId)`。
- `canUndoProject(projectId)`。
- `canRedoProject(projectId)`。

恢复项目时必须使用 `skipHistory`，避免撤销操作再次生成历史。

### 搜索服务

新增纯函数搜索模块，输入当前项目和关键词，输出分组搜索结果。UI 只负责渲染和跳转。

搜索模块不依赖 React，不访问浏览器 API，便于单测覆盖。

### 布局服务

图谱布局规则抽到纯函数模块。输入项目、图谱类型和布局视图，输出节点位置映射。

现有自动布局可以复用或迁移到该模块中。GraphWorkbench 负责预览、应用和保存，不直接承担布局算法细节。

## 错误处理

撤销或重做时，如果目标项目不存在，操作应安全返回并禁用按钮。

保存失败不应破坏历史栈。用户可以继续撤销、重做或编辑；新的成功保存会更新状态。

搜索模块遇到缺失字段时按空字符串处理，不抛出 UI 错误。

布局模块遇到空图或缺少时间字段时返回稳定的基础布局。无法计算某类分组时降级到自由或默认网格布局。

## 测试

单元测试覆盖：

- 记录历史、撤销、重做。
- 撤销后新编辑清空 redo。
- `skipHistory` 不记录历史。
- 同一 `mergeKey` 的短时间操作合并。
- 每项目历史隔离和 50 条上限。
- 保存状态成功、失败、离线、撤销后保存。
- 搜索实体、事件、关系、资料、笔记。
- 空关键词、大小写不敏感、中文子串匹配。
- 不同图谱布局视图输出稳定位置。

集成验证覆盖：

- `npm test`。
- `npm run build`。

如果本次实现触及 Electron 启动、打包或桌面专有路径，再运行 `npm run release:check`。

## 成功标准

- 用户可以通过按钮和快捷键撤销、重做当前项目主要编辑。
- 删除实体、事件或关系后，撤销能恢复数据。
- 应用图谱布局后，一次撤销能恢复原布局位置。
- 顶部能准确显示保存中、已保存、离线保存和保存失败。
- 全局搜索能查到当前项目的实体、事件、关系、资料和笔记，并能跳转。
- 图谱能切换布局视图，并在用户确认后应用位置。
- 相关单测、构建通过。
